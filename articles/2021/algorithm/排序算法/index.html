排序算法

发表于2021-02-15|更新于2021-07-06|笔记编程
|字数总计:2.7k|阅读时长:9分钟|阅读量:

注意：

本文章描述了常见几种排序算法的思想、实现、复杂度、稳定性及其性能情况，使用Java实现。

为了直观观察各个排序之间的区别，使用compare方法和swap方法分别封装了比较和交换的代码。compare的返回值就是两个参数的差。

冒泡排序

作为经典+入门排序算法老大哥，冒泡排序还是有必要整一波的qwq

思想

冒泡排序的思想就是通过对比相邻的两个元素，并交换两个元素的位置使其有序。

冒泡排序的名字由来也正是这样，大的元素不断冒向序列的两端，就像泡泡从水底冒出来一样。

实现

只要理解了冒泡排序的原理实现就很简单辣！

第一层for循环遍历除最后一个元素以外的每个元素，第二层for循环遍历第一层循环中的后一个元素，并将这两个元素比较，如果前一个元素大于后一个则将两个元素交换位置。

复杂度

冒泡排序的时间复杂度为$O(n^2)$。

稳定性

冒泡排序的是稳定的，由于只有大于前一个元素才会进行交换，所以相同大小的元素相对位置不会改变。

性能

入门的排序算法，思想简单粗暴，大量交换，性能肯定…差

测试结果如下：

1万个随机数据测试

【BubbleSort】

稳定性：true 耗时：0.457s(457ms) 比较：4999.50万 交换：2409.01万

选择排序

思想

选择排序的思想可以看做是从冒泡排序的基础上发展而来的。选择排序是每次遍历所有未排序的元素，将最大值与末尾元素交换位置；或者将最小值与首位元素交换位置。

实现

这个算法选择的方式是遍历出最大值并与末尾元素交换位置。

第一层for循环遍历每一个元素，第二层for循环则是从未排序部分寻找最大值，然后将最大值与未排序部分的末尾元素交换位置。

复杂度

选择排序的时间复杂度为$O(n^2)$。

稳定性

选择排序是不稳定的。

例如：{6, 3, 6, 4}

找到第一个6并与4交换时两个6的相对位置会改变，这种情况可以将compare的条件从>改为>=来保持稳定性，但在一些特例下选择排序仍然不是稳定的。

例如：{6, 3, 6, 4, 4}

如果将compare的条件改为>=，在这里栗子中，虽然不会改变两个6的相对位置，但是两个4的相对位置会因为第二个6与第二个4的交换而改变。

所以选择排序是不稳定的，不管如何优化，在需要保证稳定性的场景下应当不选择选择排序，因为数据总是出乎意料的。

性能

选择排序相比于冒泡排序的性能有所提升，因为选择排序将冒泡排序中「冒泡」这一过程替换成了一次交换。

测试结果如下：

1万个随机数据测试

【BubbleSort】

稳定性：true 耗时：0.463s(463ms) 比较：4999.50万 交换：2410.92万

【SelectionSort】

稳定性：false 耗时：0.2s(200ms) 比较：4999.50万 交换：1.00万

插入排序

插入排序咱觉得是一个很重要并且很强的排序了（为什么这么说呢，因为它使用的思想很简单，但是性能基本能吊打上面所说的两种排序）

思想

插入排序的思想类似于打牌时不断抓牌这一过程。插入排序是将每一个元素插入到一个已排序好的序列中（如果序列完全乱序则认为第一个元素是已经排序好的序列）。

实现

复杂度

插入排序的时间复杂度为$O(n^2)$。

稳定性

插入排序是稳定的。

性能

1万个随机数据测试

【BubbleSort】

稳定性：true 耗时：0.271s(271ms) 比较：4999.50万 交换：2432.50万

【SelectionSort】

稳定性：false 耗时：0.107s(107ms) 比较：4999.50万 交换：1.00万

【InsertionSort】

稳定性：true 耗时：0.178s(178ms) 比较：2508.82万 交换：2507.82万

可以看到插入排序的比较次数比选择排序要少很多，但是交换甚至比冒泡排序还要多，所以导致整体的效率反而比选择排序还要低。

插入排序的优化

优化#1

上面提到了交换的次数甚至比冒泡排序还要多，所以我们可以将插入排序的交换改为一次整体的插入，这样就可以减少交换次数，提高效率。

这里将一次次的swap换成了元素的移动，当然在循环移动之前，先保存好要待插入元素，然后在已排序的区间依次寻找这个元素待插入的位置。

如果当前元素比待插入的元素大，则将当前元素向后移动，直到遇到不大于待插入元素为止。

【InsertionSort1】

【InsertionSort2】

稳定性：true 耗时：0.128s(128ms) 比较：2503.93万 交换：0

可以看到优化了很多，此时性能已经直逼不稳定的选择排序。

优化#2

仔细观察上面的优化，发现在移动比待插入元素大的元素时，由于这个位置是已经排序好的，所以实际上是在进行数组的整体向后移动，了解数组在内存中的形式的话，我们就发现可以通过内存块拷贝的方式来移动数组。

当然拷贝内存这是C++等较低级的语言才能做的事，Java等高级语言为了内存安全是不允许这么做的，不过JDK还是为咱们准备了一个系统方法来拷贝数组数据。

这个系统方法就是 System.arraycopy()

这段代码咱们先把待插入元素给存到tmp变量中（在这里咱写成了泛型，可根据实际排序的数据修改），然后从开始遍历元素寻找这个元素应当被插入的位置，这个位置就是p。

再使用系统方法 System.arraycopy() 将位置p后面的元素都往后移动一位，然后将tmp赋值到p位置即可~

这次的优化没有改变比较和交换的次数，但是在移动数组这个操作上使用了更加偏底层更加高效的方法，所以大幅提升了性能。

优化#3

对于优化#2居然还有方法进一步优化，没想到吧？

在优化#2中提到了由于是向已经排好序的部分插入，所以可以使用整体的内存拷贝。不得不说这个「已排序」是真的有用，咱们还可以利用这一点加上二分搜索来优化「寻找待插入位置」这一操作！

关于优化后代码的复杂度

值得注意的是优化后的代码复杂度有变化，至于具体的复杂度是多少呢，咱也不知道233333

不仅仅是时间复杂度有变，空间复杂度也有变噢，因为 System.arraycopy() 这个方法是开辟了临时数组的。

下面是Java官方对这个方法解释的原文。

If the src and dest arguments refer to the same array object, then the copying is performed as if the components at positions srcPos through srcPos+length-1 were first copied to a temporary array with length components and then the contents of the temporary array were copied into positions destPos through destPos+length-1 of the destination array.
